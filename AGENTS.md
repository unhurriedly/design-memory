# 设记 Design Memory：Agent 项目入口

本文件是 Codex 及其他代码 Agent 进入本项目后的首要说明。目标是让 Agent 能够可靠地浏览、维护和调用设记资源库，同时保留项目的完整离线能力。

## 首次读取顺序

开始工作前按任务需要读取：

1. `AGENTS.md`：确认项目边界、数据位置和操作模式。
2. `README.md`：了解产品用途、本地启动方式、离线范围和目录结构。
3. `SKILL.md`：当任务涉及分析参考网站、复现组件、更新资源或验证设计证据时，完整阅读并遵循其中的方法。
4. `data/resources.js`：搜索目标标题、来源、标签和同类资源，确认是更新现有记录还是新增资源。
5. `components/<asset-slug>/`：只读取本次目标对应的独立预览和交互脚本。
6. `app.js`、`index.html`、`styles.css`：仅当任务涉及资源库界面、加载逻辑、搜索筛选或全局视觉时读取。

不要为了理解单条资源而一次性重构或拆分整个数据文件。

## 项目事实

- 项目是原生 HTML、CSS 和 JavaScript 静态网站，没有 npm、构建步骤或后端。
- `data/resources.js` 是预置资源的唯一事实来源。正式资源必须写入这里，不能只保存在浏览器 `localStorage` 中。
- 收藏和最近浏览属于用户本地状态，不是可分享的资源数据。
- 项目需要保持离线可运行。字体、图片、截图和组件所需媒体优先保存在项目内。
- `index.html` 使用普通脚本顺序加载 `data/resources.js` 和 `app.js`，以支持直接打开；不要重新引入只能通过 ES Modules 或构建工具运行的入口。
- 资源详情中的外部来源链接用于溯源，可以保留；未点击时不得成为页面运行依赖。
- 当前正式资源状态使用 `ready`。不要恢复已经移除的“待分析内容”用户流程或 `pending` 资产入口，除非用户明确要求。

## 目录与职责

```text
data/resources.js                  资源元数据、设计说明、证据与可复制代码
components/<asset-slug>/demo.html 独立组件预览
components/<asset-slug>/demo.js   复杂交互逻辑，可选
assets/components/<asset-slug>/   组件本地图片与媒体
assets/references/                原站或用户提供的参考截图
assets/fonts/                     离线字体文件
assets/brand/                     设记品牌资产
app.js                            搜索、筛选、详情、收藏和最近浏览逻辑
styles.css                        资源库外壳的全局样式
index.html                        资源库入口与图标定义
README.md                         面向使用者的项目说明
SKILL.md                          网站分析、复现和入库规范
```

路径约定：

- `data/resources.js` 中的 `preview` 和 `referenceImage` 从项目根目录书写，例如 `./components/example/demo.html`。
- 独立预览位于 `components/<asset-slug>/`，引用项目资产通常使用 `../../assets/...`。
- 资源详情里的可复制代码面向项目根目录使用，若依赖本地字体或图片，要在说明中指出需要同时复制的资产。

## 三种工作模式

### 1. 浏览和回答

当用户询问已有资源、设计参数或实现方式时：

1. 在 `data/resources.js` 搜索自然语言标题、标签、来源和用途。
2. 阅读对应记录的 `description`、`usage`、`implementation`、`tokens`、`evidence` 和 `code`。
3. 需要确认运行效果时，再打开对应 `components/<asset-slug>/`。
4. 回答时区分原站实测、代码推断和本地复现，不把推断描述成原站源码。

只读问题不授权修改项目文件。

### 2. 新增或更新资源

当用户发送网站、截图或开源项目并要求加入设记时：

1. 完整遵循 `SKILL.md` 的取证、判断、复现和验证流程。
2. 先查重；相同机制或相同来源的已有资源优先更新，不重复新增。
3. 在 `components/<asset-slug>/` 制作可独立运行的预览。
4. 将必要媒体保存到 `assets/components/`、`assets/references/` 或 `assets/fonts/`。
5. 在 `data/resources.js` 新增或更新正式记录，提供自然语言说明、精确参数、证据和完整代码。
6. 保持资源库离线运行，不依赖原站字体 CDN、favicon 服务、脚本、认证或路由。

### 3. 在其他前端项目中调用资源

当 Agent 在另一个项目中使用设记辅助设计和开发时：

1. 先确认设记目录对当前 workspace 可读；不要假设另一个项目能自动访问本目录。
2. 根据目标项目的行业、页面区域、技术栈、视觉气质和交互目标，在 `data/resources.js` 筛选候选资源。
3. 组合资源前说明选择理由，例如字体系统、卡片、布局和动效之间如何配合。
4. 阅读 `implementation`、`tokens`、`evidence` 和 `code`，再查看对应独立预览。
5. 识别并复制必要的字体、图片和脚本；不要留下指向设记目录的脆弱绝对路径。
6. 将原生代码适配为目标项目已有的 React、Vue、原生 HTML 或组件系统，沿用目标项目的变量、无障碍和响应式约定。
7. 不覆盖目标项目已有设计系统，也不机械复制与业务不匹配的文案和品牌内容。
8. 在目标项目完成桌面端、移动端、键盘操作、动效降级和控制台验证。

默认情况下，跨项目调用只修改目标项目，不反向修改设记；发现设记记录有错误时，先向用户说明再更新源资源。

## 修改权限边界

一般资源任务可以修改：

- `data/resources.js`
- 本次资源对应的 `components/<asset-slug>/`
- 本次资源对应的 `assets/components/`、`assets/references/` 或 `assets/fonts/`
- 与本次工作流直接相关的 `README.md`、`SKILL.md` 或 `AGENTS.md`

只有用户明确要求资源库产品界面或全局能力变化时，才修改：

- `app.js`
- `styles.css`
- `index.html`
- 启动脚本和整体目录结构

不要修改或删除：

- 与当前任务无关的已有资源、预览和用户文件。
- 浏览器 `localStorage` 来伪造项目数据更新。
- 原始参考截图，除非新文件已验证且用户要求替换。
- 用户未授权的目标项目、外部仓库、线上页面或发布配置。

工作区可能存在未提交改动。先查看 `git status`，保留并绕开与当前任务无关的变化，不执行破坏性 Git 命令。

## 资源记录最低要求

每条正式资源至少应包含：

- 稳定且唯一的 `id`
- 完整的 `source`、`title`、`type`、`status` 和 `updatedAt`
- 可运行的 `preview`
- 清晰的 `description`、`usage` 和 `implementation`
- 可检索的自然语言 `tags`
- 精确的设计 `tokens`
- 可复查的 `evidence`
- 可复用的 `code.html`、`code.css` 和 `code.js`

资源类型只能沿用：`typography`、`card`、`layout`、`interaction`、`graphic`。

资源说明必须让人或 Agent 在不查看原站的情况下理解：它是什么、适合何时使用、关键参数是多少、如何实现、哪些内容可以替换、有哪些依赖和限制。

## 离线要求

- 页面和组件运行不得请求外部字体、favicon、图片或必须执行的第三方脚本。
- 已存在的字体优先复用 `assets/fonts/`，不要重复下载同一字重。
- 外部 URL 可以出现在来源、证据和说明文字中，但不能作为预览成功运行的前提。
- 开源依赖若无法用少量本地文件可靠保存，应记录依赖和集成方法，不把联网安装伪装成离线实现。
- 新增本地资产后检查文件格式、路径、体积和桌面/移动端加载结果。

## 验证要求

完成修改后，至少执行：

```bash
node --check data/resources.js
node --check app.js
```

若修改了独立 `demo.js`，也对它运行 `node --check`。随后：

1. 启动静态服务器，例如 `python3 -m http.server 4174 --bind 127.0.0.1`。
2. 验证资源库首页、搜索、筛选、目标详情和独立预览。
3. 检查桌面端与约 390px 移动端，无非预期横向滚动、遮挡、空白 iframe 或文字溢出。
4. 实际触发 hover、点击、拖拽、切换、滚动或状态动画等核心交互。
5. 检查控制台错误和运行时外网请求。
6. 确认字体、图片和参考截图均从本地路径加载。
7. 运行 `git diff --check`，并再次查看 `git status`，确保没有夹带或覆盖无关改动。

无法执行某项验证时，在交付说明中明确指出，不要把未验证结果写成已通过。

## 交付说明

完成后向用户简要说明：

- 新增或更新了哪些资源和文件。
- 原站事实、本地复现和可修改参数中的关键结论。
- 执行了哪些验证，以及仍存在的限制。
- 是否存在未提交、未推送或与本次任务无关的工作区变化。

除非用户明确要求，不自动提交、推送、发布或删除文件。
